Cache Schema

On-device data in Room/SQLite is the source of truth for a single user. Once a workout is synced to the backend, leaderboards, trending exercises, and other cross-user aggregates are served from an in-memory key-value cache sitting in front of the backend's relational store. The cache holds hot, frequently-read data with short TTLs; SQL stays the durable system of record.

:::note Server-side — future backend phase Everything on this page lives server-side and belongs to a future backend phase: leaderboards and cross-user stats require a central Valkey instance fed through a sync API that is not defined yet. The on-device database (DB schema) remains local and single-user until that phase ships. :::

Data flow​

Android (SQLite / Room) → REST API (backend) → in-memory cache → leaderboards / global stats

Sync is triggered on session complete, on a new PR, and on app open (background refresh).

Sync strategy (SQL → cache)​

TriggerAction
workout_session completedPush volume score · update user:{id}:stats · check streak
Strength PR insertedUpdate pr:exercise:{id}:1rm and pr:exercise:{id}:volume
Endurance PR insertedUpdate pr:exercise:{id}:distance and pr:exercise:{id}:pace
session_exercise insertedIncrement exercise:{id}:use_count · bump exercises:trending
App opens (background)Refresh user:{id}:stats if its TTL has expired

pace_per_km is computed as duration / (distance / 100000) — distance is stored in centimeters (see storage conventions) — before being written. All cache writes are fire-and-forget — the relational store is the source of truth, and a failed cache write never blocks the request.

Key design decisions​

  • Cache-aside, not cache-through — reads hit the cache first; on a miss (expired TTL) they fall back to SQL and re-populate the cache. The cache never holds data SQL doesn't also have.
  • Every key has a TTL except the persistent counters (exercise:{id}:use_count), so a cold cache always self-heals from SQL without manual invalidation.
  • Pace is ranked ascending, everything else descending — leaderboard reads need to know this per key, since a lower pace is a better result.
  • One sorted set per exercise for leaderboards, rather than one global structure, keeps per-exercise ranking queries O(log n) instead of requiring a filtered scan.
  • Writes are fire-and-forget — the cache layer is an optimization for read-heavy endpoints (leaderboards, trending), not a dependency for write correctness.
