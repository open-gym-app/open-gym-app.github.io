Database Schema

OpenGym uses SQLite via Room (Android Jetpack). The schema is fully normalized and enforces foreign keys throughout.

The tables listed below describe the full long-term data model: most of them (gamification, marketplace, gym map, plans…) are not implemented yet, and names may differ between the two (e.g. session → workout_session, exercise_set → workout_set, exercise_muscle → exercise_muscle_group). When in doubt, the diagram is authoritative for what exists today. :::

Storage conventions​

No floating-point columns anywhere — every measure is a 64-bit integer (INTEGER is stored on 8 bytes in SQLite; map to Long in Room). Column names never carry units.

MeasureColumnsStored as
Weightweight, default_weight, estimated_1rmmilligrams
Distancedistancecentimeters
Time spansduration, rest_after, rest_duration, default_rest, paceseconds
RPErpevalue × 10 (65 = RPE 6.5), range 10–100
Timestamps*_atUnix epoch milliseconds; date-only fields (created_at (date)) store the day at midnight UTC

All unit conversions — including imperial ↔ metric display — happen in the frontend; the database only ever sees these canonical units.

Tables​

Core​

TablePurpose
user_profileSingle-row profile: name, goals, fitness level, preferred unit system
body_measurementTime-series body weight, body fat %, and per-body-part measurements

Exercise library​

TablePurpose
exerciseMaster list of exercises: name, type, primary/secondary muscles, media URL
muscleMuscle reference table used for body-map highlights
exercise_muscleMany-to-many join: exercise ↔ muscle (with is_primary flag)

Workout sessions​

TablePurpose
sessionA single workout: start/end timestamps, notes, resume_token for mid-session persistence
session_exerciseOrdered exercises within a session
exercise_setIndividual set: reps, weight, duration, distance, rest duration, notes

Plans & templates​

TablePurpose
workout_planNamed plan (Beginner / Intermediate / Advanced / custom)
plan_dayA day within a plan
plan_day_exerciseExercises assigned to a plan day
session_templateSaved session layout for one-click re-use
template_exerciseExercises within a template

Performance & records​

TablePurpose
personal_recordOne row per PR type per exercise: max weight, max reps, 1RM estimate, real 1RM
performance_historyRaw per-set history used to compute progression graphs

Gamification​

TablePurpose
xp_eventAppend-only log of XP-earning events
achievementCatalogue of achievable milestones
user_achievementUnlocked achievements with unlock timestamp
loot_itemCatalogue of loot: titles, pet eggs, pets, mounts
user_lootLoot owned by the user

Marketplace​

TablePurpose
productAggregated product listings with shop, price, affiliate flag
product_reviewCommunity reviews: 1–5 star rating + written body
user_wishlistProducts saved to the user's wishlist/shopping list

Gym map​

TablePurpose
gymDiscovered gyms: name, coordinates, visit count
gym_reviewAdvice, comments, and star rating per gym

Key design decisions​

  • Foreign keys are enforced (PRAGMA foreign_keys = ON at connection open).
  • Indexes exist on all frequently queried columns: session.started_at, exercise_set.session_exercise_id, performance_history.exercise_id, and xp_event.created_at.
  • resume_token on session stores a lightweight JSON snapshot so an in-progress workout survives process death.
  • Single-user model — no user_id scattered across tables; user_profile is always a single row.
  • Append-only XP log — xp_event is never mutated; total XP is always computed from the sum of the log.
